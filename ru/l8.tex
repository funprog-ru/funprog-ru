\frame{
  \frametitle{Сохранение локальных переменных}
  % \heading{Storing local variables}

  Вспомним наше определение факториала:
  % Recall our definition of the factorial:

  \ExampleOldFact

  Вызов {\tt fact 6} порождает другой вызов {\tt fact 5} (и так
  далее), но вычислителю нужно сохранять старое значение {\tt 6},
  чтобы в дальнейшем произвести умножение.
  % A call to {\tt fact 6} causes another call to {\tt fact 5} (and
  % beyond), but the computer needs to save the old value {\tt 6} in order
  % to do the final multiplication.

  Поэтому, локальные переменные функции, в нашем случае {\tt n},
  не могут храниться в фиксированном месте, потому что при каждом вызове
  функции нужны выделенные копии переменных.
  % Therefore, the local variables of a function, in this case {\tt n},
  % cannot be stored in a fixed place, because each instance of the function needs
  % its own copy.

  Вместо этого, при каждом вызове функции выделяется место в {\em стеке}.
  % Instead each instance of the function is allocated a frame on a {\em stack}.

  Этот способ в ML схож с другими языками, которые поддерживают
  рекурсию, включая C.
  % This technique is similar in ML to most other languages that support recursion,
  % including C.

}

\frame{
  \frametitle{Хвостовая рекурсия}
  % \heading{A tail recursive version}

  Теперь, в противопоставление старому определению, рассмотрим следующее
  определение функции факториала:
  % Now, by contrast, consider the following implementation of the factorial
  % function:

  \ExampleNewFact

  Рекурсивный вызов~-- итоговое выражение, после него никаких вычислений
  не происходит; оно не является подвыражением, частью другого
  выражения.

  Подобные вызовы называются {\em хвостовыми вызовами}, потому что самые
  последние в теле функции.

  А функция, в которой все рекурсивные вызовы суть хвостовые, называется
  функцией с {\em хвостовой рекурсией}.

  % The recursive call is the whole expression; it does not occur as a proper
  % subexpression of some other expression involving values of variables.

  % Such a call is said to be a {\em tail call} because it is the very last thing
  % the calling function does.

  % A function where all recursive calls are tail calls is said to be {\em tail
  %   recursive}.
}

\frame{
  \frametitle{Зачем нужна хвостовая рекурсия?}
  % \heading{Why tail recursion?}

  Компилятор достаточно умён, чтобы понять, что для функций с хвостовой
  рекурсией можно использовать одно и тоже место в памяти для хранения
  локальных переменных при рекурсивных вызовах.
  % If a function is tail recursive, the compiler is clever enough to realize that
  % the same area of storage can be used for the local variables of each instance.

  Мы избежим переполнения стека.
  % We avoid using all that stack.

  Дополнительный аргумент {\tt x} функции {\tt tfact} называется {\em
    аккумулятором}, потому что он накапливает промежуточные результаты по
  ходу рекурсивных вызовов и, в конце концов, возвращается как значение
  функции.
  % The additional argument {\tt x} of the {\tt tfact} function is
  % called an {\em accumulator}, because it accumulates the result as the recursive
  % calls stack up, and is then returned at the end.

  Определение функций таким образом, является общепринятым способом
  написания функций с хвостовой рекурсией.
  % Working in this way, rather than modifying the return value on the way back up,
  % is a common way of making functions tail recursive.

  В этом смысле, реализация функции с хвостовой рекурсией, использующая
  аккумуляторы соответствует итеративной версии с использованием
  присваиваний и while-циклов.
  % In a sense, a tail recursive implementation using accumulators corresponds to
  % an iterative version using assignments and while-loops.

  Единственная разница это то что мы явно передаём состояние.
  % The only difference is that we pass the state around explicitly.
}

\frame{
  \frametitle{Минимизация операций cons (1)}
  % \heading{Minimizing consing (1)}

  Память, используемая конструкторами типа (<<cons-ячейки>>), не
  выделяется и не освобождается таким же простым способом как и на
  стеке.
  % The space used by type constructors (<<cons cells>>) is not allocated and
  % deallocated in such as straightforward way as stack space.

  В общем случае достаточно сложно понять когда определённая cons-ячейка
  занята и когда её можно использовать повторно.  Например:
  % In general, it is difficult to work out when a certain cons cell is in use, and
  % when it is available for recycling. For example in:

  \ExampleGCCons

  \noindent cons-ячейку можно сразу же освобождать.  Однако если {\tt
    l} был передан различным функциям, то во время компиляции
  невозможно определить когда эта cons-ячейка больше не нужна.
  % \noindent the cons cell can be reused immediately. However if {\tt l} is
  % passed to other functions, it is impossible to decide at compile-time when the
  % cons cell is no longer needed.

  Поэтому, в функциональных языках необходимо время от времени проверять
  какая часть памяти больше не используется и освобождать её для
  дальнейших вычислений.  Такой процесс называется {\em сборка мусора}.
  % Therefore, space in functional languages has to be reclaimed by analyzing
  % memory periodically and deciding which bits are needed. The remainder is then
  % reclaimed. This process is known as {\em garbage collection}.

}

\frame{
  \frametitle{Ссылки (1)}
  % \heading{References (1)}

  В ML есть настоящие переменные, к которым можно присваивать, и
  выражения могут в качестве побочного эффекта изменять значения этих
  переменных.
  % ML does have real assignable variables, and expressions can, as a side-effect,
  % modify the values of these variables.

  Доступ к переменным осуществляется с помощью {\em ссылок} (указателей,
  говоря языком C), и ссылки в свою очередь рассматриваются в ML как
  обычные значения.
  % They are explicitly accessed via {\em references} (pointers in C parlance) and
  % the pointers themselves behave more like ordinary ML values.

  Чтобы создать новую ячейку памяти и инициализировать её значением
  {\tt x} нужно записать {\tt ref x}.  Это выражение
  возвращает соответствующую ссылку, т.е. указатель на вновь созданную
  ячейку.
  % One sets up a new assignable memory cell with the initial contents {\tt
  %   x} by writing {\tt ref x}. This expression returns the corresponding
  % reference, i.e. a pointer to it.

  Изменяют содержимое ячейки через указатель.
  % One manipulates the contents via the pointer.

  Этот подход довольно похож на использование указателей в C: например
  когда требуется использовать <<изменяемые параметры>>~-- параметры,
  для которых допустимо изменение значений после вычисления функции,
  используется передача параметров по указателю.
  % This is quite similar to C: here one often simulates <<variable parameters>> and
  % passes back composite objects from functions via explicit use of pointers.

  Содержимое ссылки извлекается с помощью оператора разыменования {\tt
    !}.  А чтобы изменить значение, используется {\tt :=}.
  % To get at the contents of a {\tt ref}, use the dereferencing
  % (indirection) operator {\tt !}. To modify its value, use {\tt
  %   :=}.
}
